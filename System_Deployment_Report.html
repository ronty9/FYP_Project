1. System Deployment Overview

PawScope is a cross-platform mobile application built with Flutter that provides pet owners with AI-powered breed identification and skin disease detection, alongside comprehensive pet health management features. The system deployment involves three major components:

Component Technology Deployment Target
Mobile Application Flutter (Dart) Android (APK/AAB), iOS (IPA)
AI Backend Server Python, FastAPI, TensorFlow/Keras Google Cloud Run (Docker)
Cloud Services Firebase Auth, Firestore, Storage, Cloud Functions Google Firebase Platform

1.2 Deployment Tasks Completed

The following implementation tasks were carried out during the deployment phase:

1.2.1 File Creation and Conversion

Task Description
AI Model Packaging TensorFlow/Keras models (species_best.keras, breed models, disease models) were packaged from the training environment (lib/AI/models/) into the Docker container at /app/models/.
Docker Containerisation The AI backend was containerised using a Dockerfile based on python:3.11-slim with all dependencies (tensorflow, pillow, fastapi, firebase-admin, etc.) installed via requirements.txt.
Firebase Configuration Platform-specific configuration files were generated: google-services.json (Android), GoogleService-Info.plist (iOS/macOS), and firebase_options.dart (Dart).
Security Rules Firestore and Storage security rules were authored to enforce per-user data isolation (users can only read/write their own data).
Cloud Functions Two serverless functions (resetPassword and sendOtp) were deployed to the asia-southeast1 region using Firebase Cloud Functions v2, with API keys stored as Firebase Secrets.
Application Build Flutter release builds were generated for Android (APK/AAB) and iOS (IPA) using flutter build apk --release and flutter build ios --release.

1.2.2 Changeover Strategy

A Direct Changeover (also known as “Big Bang”) strategy was adopted for the initial deployment. Since PawScope is a new system without a predecessor, there was no legacy system to run in parallel. The deployment proceeded as follows:

  1. Backend First — The AI backend was deployed to Google Cloud Run and verified with the /health endpoint.
  2. Firebase Services Second — Firestore rules, Storage rules, and Cloud Functions were deployed and tested.
  3. Mobile App Last — Once all backend services were confirmed operational, the mobile application was distributed to test devices and then released.

2. System Backup and Risk Management

2.1 Backup Procedures

Since PawScope relies on cloud-managed services, backup procedures are designed around each component:

Component Backup Method Frequency Retention
Firestore Database Firebase automated daily backups + manual export via gcloud firestore export to a Cloud Storage bucket Daily (automated), On-demand (manual before major changes) 30 days
Firebase Storage (pet photos, scan images, profile photos) Object versioning enabled on the Cloud Storage bucket; cross-region replication configured Continuous (versioning) Current + 3 previous versions
AI Models Model files versioned in Git repository (lib/AI/models/); production copies stored in the Docker image registry (Google Artifact Registry) Per deployment All tagged versions retained
Source Code Git version control with remote repository; feature branching workflow Continuous Full history
Cloud Functions Source code in /functions/ directory under Git; deployed versions retained by Firebase Per deployment All versions retained
Firebase Auth Managed entirely by Firebase; user export available via firebase auth:export On-demand N/A (managed service)

Pre-Changeover Backup Checklist

Before each deployment or major changeover, the following backup steps are executed:

  1. gcloud firestore export gs://pawscope-3b1be-backups/pre-deploy-$(date +%Y%m%d) — Export full Firestore database.
  2. Verify Storage bucket versioning is active.
  3. Tag the current Git commit: git tag -a pre-deploy-v1.x -m "Pre-deployment backup".
  4. Record the current Cloud Run revision ID for potential rollback.

2.2 Risk Assessment and Mitigation

# Risk Impact Likelihood Mitigation
1 AI Backend Downtime — Cloud Run service becomes unavailable High — Breed/disease scanning features unusable Low Cloud Run auto-scaling with min-instances set to 1; health-check endpoint (/health) monitored; the app displays a user-friendly error message (“AI service temporarily unavailable, please try again”) with retry option.
2 Data Loss — Firestore or Storage data corruption Critical — Loss of user data, pet profiles, scan records Very Low Daily automated backups; Firestore point-in-time recovery; Storage object versioning; security rules prevent unauthorised writes.
3 API Key Exposure — SendGrid or Firebase keys leaked High — Potential abuse of email services or Firebase resources Low API keys stored as Firebase Secrets (not in source code); Application Default Credentials used for backend authentication; .gitignore configured to exclude sensitive files.
4 Model Accuracy Degradation — AI models produce incorrect results over time Medium — Incorrect breed/disease identification Medium Admin module allows review of analysis records; user feedback mechanism in place; model versioning enables rollback to previous models.
5 Failed Deployment — New version introduces breaking changes High — App crash or critical feature failure Medium Git tagging for rollback; Cloud Run traffic splitting (canary deployment); staged rollout on app stores; automated unit tests (flutter test) run before deployment.
6 Firebase Service Outage — Google Firebase platform experiences downtime Critical — Entire app non-functional Very Low Firebase provides 99.95% SLA; local caching with shared_preferences for session persistence; offline-first patterns where feasible.

2.3 Rollback Procedure

In the event of a failed deployment:

  1. Cloud Run Backend: Roll back to the previous revision using gcloud run services update-traffic petscan-ai --to-revisions=<previous-revision>=100.
  2. Cloud Functions: Redeploy from the previously tagged Git commit using firebase deploy --only functions.
  3. Mobile App: If released to app stores, submit a hotfix or use staged rollout to halt distribution. For internal testing, redistribute the previous APK/IPA.
  4. Firestore Rules: Revert rules from Git and redeploy using firebase deploy --only firestore:rules.

3. On-site Setup

3.1 Pre-Setup Preparations

Before setting up PawScope on the client’s infrastructure, the following preparations were completed:

# Preparation Task Details
1 Firebase Project Setup Created Firebase project pawscope-3b1be with Blaze (pay-as-you-go) billing plan to enable Cloud Functions and Cloud Run.
2 Google Cloud Project Configuration Enabled Cloud Run API, Cloud Build API, Artifact Registry API, and Firebase Admin SDK in the Google Cloud Console.
3 Domain & Region Selection Selected asia-southeast1 (Singapore) as the primary region for Cloud Functions and Cloud Run to minimise latency for the target user base.
4 Authentication Providers Configured Email/Password and Google Sign-In authentication providers in Firebase Auth console.
5 SendGrid Account Created a SendGrid account and configured a verified sender email (pawscope1@outlook.com) for OTP email delivery.
6 Development Environment Ensured Flutter SDK (≥3.9.2), Dart SDK, Python 3.11, Docker, Firebase CLI, and Google Cloud CLI were installed on the development machine.
7 Test Devices Prepared Android devices (Android 12+) and iOS devices (iOS 15+) with developer mode enabled for on-device testing.

3.2 Setup Procedures

The on-site setup follows a sequential process to ensure each layer is operational before proceeding:

Phase 1: Firebase Services Deployment (Day 1)

Step 1: firebase login
Step 2: firebase use pawscope-3b1be
Step 3: firebase deploy --only firestore:rules
Step 4: firebase deploy --only storage
Step 5: firebase functions:secrets:set SENDGRID_API_KEY
Step 6: firebase deploy --only functions
Step 7: Verify Cloud Functions in Firebase Console

Phase 2: AI Backend Deployment (Day 1–2)

Step 1: cd backend
Step 2: docker build -t petscan-ai .
Step 3: docker run -p 8080:8080 petscan-ai  (local verification)
Step 4: curl http://localhost:8080/health  (verify health endpoint)
Step 5: gcloud run deploy petscan-ai \
          --source ./backend \
          --region asia-southeast1 \
          --allow-unauthenticated \
          --memory 2Gi \
          --cpu 2 \
          --min-instances 0 \
          --max-instances 3
Step 6: Verify deployment at Cloud Run URL /health
Step 7: Test /predict and /predict-disease endpoints with sample images

Phase 3: Mobile Application Build & Distribution (Day 2–3)

Step 1: flutter pub get
Step 2: flutter test  (run automated unit tests)
Step 3: flutter build apk --release  (Android)
Step 4: flutter build ios --release  (iOS)
Step 5: Distribute APK via internal testing track / Firebase App Distribution
Step 6: Submit iOS build via TestFlight for internal testing
Step 7: Perform end-to-end testing on physical devices

Phase 4: Verification & Go-Live (Day 3–4)

Test Expected Result Status
User registration (email + Google) Account created, OTP email received ✓
Login / persistent session User remains logged in after app restart ✓
Pet profile CRUD Create, read, update pet profiles with photo upload ✓
AI Breed Identification Upload photo → species + top-5 breed results returned ✓
AI Skin Disease Detection Upload photo → species + disease predictions returned ✓
Schedule management Create, view, edit calendar events with reminders ✓
Password reset (OTP flow) OTP sent via email → password updated via Cloud Function ✓

5. Follow-up Plan

5.1 Follow-up Objectives

The follow-up phase aims to verify system reliability, gather user feedback, and address any issues that emerge post-deployment. The plan spans 8 weeks after the initial go-live date.

5.2 Follow-up Schedule

Week Activity Method Responsible
Week 1 Daily monitoring — Check Cloud Run logs, Firebase Console for errors, monitor Cloud Function execution Firebase Console, Google Cloud Monitoring Developer
Week 2 User feedback collection — Gather initial impressions and issues via in-app feedback form In-app feedback module Developer + Client
Week 3 Performance review — Analyse API response times, app load times, AI prediction latency Cloud Run metrics, Firebase Performance Developer
Week 4 Bug fix release — Address reported issues, deploy patch update to Cloud Run and app stores Git, CI/CD pipeline Developer
Week 5 User satisfaction survey — Distribute survey to assess usability and feature satisfaction Google Forms / In-app Client
Week 6 AI model accuracy review — Review analysis records, compare AI predictions with verified diagnoses Admin dashboard analysis records module Developer + Client
Week 7 System health audit — Full review of Firestore usage, Storage costs, Cloud Run scaling, security rules Google Cloud Console Developer
Week 8 Final handover meeting — Present system health report, address outstanding issues, discuss future enhancements In-person / Video call Developer + Client

5.3 Monitoring Metrics

The following key performance indicators (KPIs) will be tracked during the follow-up period:

Metric Target Monitoring Tool
API Response Time (breed/disease prediction) < 5 seconds Cloud Run logs
App Crash Rate < 1% Firebase Crashlytics (if configured)
Cloud Function Success Rate (OTP, Password Reset) > 99% Firebase Functions logs
User Registration Rate Trending upward Firestore user collection count
AI Prediction Accuracy > 85% confidence on correct results Admin analysis records
User Feedback Score ≥ 4.0 / 5.0 In-app feedback module

5.4 Issue Escalation Procedure

Level 1: Minor UI/UX issues
  → Logged in feedback system → Addressed in next scheduled patch

Level 2: Feature malfunction (e.g., scan not working, schedule not saving)
  → Immediate investigation → Hotfix deployed within 48 hours

Level 3: Critical system failure (e.g., backend down, data loss)
  → Emergency response → Rollback to last known good version
  → Root cause analysis within 24 hours
